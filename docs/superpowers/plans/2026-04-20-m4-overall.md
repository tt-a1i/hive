# M4 Overall Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the M2-delayed spec items by landing schema-backed presets/templates/app state, Layer B fallback plus Layer A environment sync, `tasks.md` watcher + UI dual-mode editing, and a real multi-workspace sidebar flow.

**Architecture:** M4 is sequenced, not monolithic. M4.1 establishes the missing data model and recovery/runtime wiring, because command presets, role templates, and app state are prerequisites for cleaner worker/workspace UX and for turning the existing Claude-specific resume code into a preset-driven engine. M4.2 and M4.3 build on that foundation: watcher/editor sync can reuse the persisted app/workspace model, and the sidebar UX can only be made durable once `app_state` exists and add-worker flows stop depending on hardcoded templates.

**Tech Stack:** Existing Node.js 22 ESM, better-sqlite3, React 19, Vitest, node-pty, ws. New dependency expected in M4.2 only: `chokidar` for `tasks.md` watching. M4.1 should avoid new runtime dependencies unless a migration helper is strictly necessary.

## Chunk 5: Layer B Wiring

### 5. Layer B ç”ںن؛§ wiring

Current code facts:
- `src/server/tasks-file.ts:15-34` already provides `readTasks` / `writeTasks`.
- `messages` already carry `user_input | send | report` in SQLite.
- `recovery-summary.ts` is absent today, so M4.1 PR-3 should recreate it intentionally instead of reviving deleted code blindly.

### Current inclination

- Reintroduce `src/server/recovery-summary.ts` as a **pure string-builder helper**, not as a store.
- Put the trigger logic in a new restart-specific module, e.g. `src/server/restart-fallback.ts`, that decides:
  1. Try Layer A using preset-driven resume.
  2. If Layer A is unavailable or fails, build a Layer B summary from `messages`, `tasks.md`, and worker state.
  3. After successful Layer A resume, inject the separate environment-sync message.
- Precedence should be:
  - Layer A success -> env-sync message only
  - Layer A unsupported / missing / failed -> Layer B summary
  - Never send both Layer A env-sync and Layer B summary for the same restart attempt.

### Why this matters

- It preserves the specâ€™s â€œLayer A first, Layer B fallbackâ€‌ model.
- It keeps `runtime-store.ts` from absorbing restart policy logic.
- It gives M4.1 PR-3 a clean testing surface for true integration tests.
